# Debrief 格式

每场盘问结束后的**独立存档**，记录本场战绩。只增不改，构成用户的成长轨迹。
存于 `debriefs/`，编号递增：`0001-<dash-case>.md`、`0002-<dash-case>.md`……

## 结构

```md
# 复盘 0001：{本场主题，如 后端项目性能优化}

- **日期**：{YYYY-MM-DD}
- **盘问强度**：{友善教练 / 专业面试官 / 压力面}
- **目标岗位**：{引用 MISSION 的岗位画像，用于解释及格线}

## 本场拷问点与裁定

### Probe Point 1：{标题}
- **问的问题**：{追问链概述}
- **四维裁定**：正确性 强/中/弱 ｜ 深度 _ ｜ 真实性 _ ｜ 表达 _
- **点评 / 纠错 / 补漏 / 建议**：{结合四维的具体反馈}

### Probe Point 2：{标题}
...

## 暴露的盲区
- {本场新发现或再次确认的知识盲区 / 经历水分}

## 简历修改建议（现实落点）
- **删 / 弱化**：{撑不住的经历}
- **强化**：{讲得透、可放大的亮点}
- **补量化**：{缺数字结果的条目，建议补什么指标}

## 下一步
- {建议复访时重点攻克的弱点；或适时建议去做真实模拟面试}
```

## 规则

- **只增不改**：每场一份新文件，不回头改旧复盘。
- 四维裁定务必写全，与弱点雷达的状态更新保持一致。
- "简历修改建议"是本 skill 的现实落点，每份 Debrief 都应给出，反哺到简历本身。
- "下一步"在弱点反复攻不下或用户准备充分时，点一句去真实模拟面试，点到为止。
